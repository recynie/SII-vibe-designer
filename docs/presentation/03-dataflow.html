数据流与迭代机制

文件即契约 · Critic 闭环评审 · Designer→Critic→迭代 循环
03 / 05 — DATA FLOW
01

Brief 落盘

RESEARCHER

WebSearch 收集主题信息 → 写 brief.md（核心信息、受众、调性）+ brand-spec.md（色板、字体、视觉方向）。下游所有 agent 共读。

02

Plan 拆解

PLANNER

读 brief + spec → 写 plan.md，列出子任务（logo / poster / copy / ui-mockup），每条标注对应 skill 和迭代策略。

03

Designer 出物

DESIGNER · SKILL 路由

根据子任务类型加载 skill prompt → 调 gen_image 出图 或 Write HTML → 落 v1.png + v1.prompt.txt。prompt 一起存盘，供 Critic 审。

04

Critic 评审

CRITIC · 5 维度

读实物（图/截图）+ 读 prompt → 5 维度 0-10 分 → 写 v1.review.md。总分 ≥35 且无单项 ≤4 则通过。

05
↻ 迭代循环 · 最多 2 轮

不通过 → Designer 读 review → 改 prompt 或改版式 → 出 v2 → Critic 再评。
2 轮仍不过 → Planner 向用户提问（人在环路的安全阀）。

06

汇总交付

PLANNER

写 final.md，列所有产物路径、评分、已知局限。两段式回复用户。

Critic 评分体系 · 5 维度

品牌契合（与 brand-spec 一致） 0-10
信息层级 0-10
视觉品味（Pentagram 信息建筑派） 0-10
反 AI slop（一票否决） 0-10
任务完成度 0-10
通过门槛：≥ 35/50 且无单项 ≤ 4
反 AI slop 项 ≤ 4 则整体不通过——紫渐变、emoji 图标、Inter 当 display、SVG 画人物、generic 拼贴卡片均判零。

不同介质 · 不同迭代方式

产出类型迭代方式上限
图（logo / 插画）改 prompt 重出2 轮
HTML（海报 / UI）截图评审 → 改样式2 轮
文案出 3 变体 → 选优改一轮1 轮

文件即上下文契约

brief.md — 主题核心信息
brand-spec.md — 色板字体方向
plan.md — 子任务清单
v?.prompt.txt — 生成 prompt
v?.review.md — Critic 评审
final.md — 交付汇总
